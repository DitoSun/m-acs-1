# M-ACS-1 Real-World Observation Charter

v0.1.0 已发布。当前阶段只做一件事：**观察真实用户怎么用，不主动加功能。**

---

## 1. Observation Discipline

### 三条纪律

```
纪律 1: 用户不反馈，就不修改。
   └─ 不主动猜测用户想要什么
   └─ 不基于假设优化
   └─ 不提前做"用户将来可能需要"的功能

纪律 2: 一个用户的问题，不一定是问题。
   └─ 同一个问题出现 3 次 → 才确认是模式
   └─ 3 个独立用户提出同一需求 → 才考虑
   └─ 10 个用户都遇到同一问题 → 立即修

纪律 3: 用户说"想要 X"，不是要做 X。
   └─ 用户说 "想要 OCR" → 其实可能是因为 PDF 打不开
   └─ 用户说 "想要多用户" → 其实可能是想和同事分享
   └─ 用户说 "想要更好的检索" → 其实可能是 chunking 有问题
   └─ 先理解用户真正的问题，再决定方案
```

### Feature Panic 预防

```
当出现以下冲动时，停下来:
────────────────────────────────
□  "这个功能很酷，加吧"            → 等 3 个用户来要
□  "竞品有这个功能"                → 我们不是竞品
□  "这个技术很新"                  → 新技术 ≠ 用户需要
□  "这个用户很重要，他的需求必须做"  → 确认是否代表多数
□  "这周末没事做，加个小功能"       → 周末应该看用户反馈
□  "用户想要，用户是对的"          → 用户描述的是问题，不是方案
```

### 时间分配

```
v0.1.x 阶段的时间:
  40% 观察用户反馈
  30% 修 Bug
  20% 改进文档
  10% 新增测试

  0% 新功能
```

## 2. Failure Prioritization

### 唯一优先级标准

```
一个 failure 的优先级 = 它影响了多少用户 × 它阻止了什么使用场景
```

### 优先级矩阵（更新版）

```
等级  定义                响应时间       示例
─────────────────────────────────────────────────────
P0    用户完全无法使用     24h           安装失败、LLM 编造、服务崩溃
P1    关键功能不可用       72h           检索不到正确条款、source 页码不对
P2    功能异常但有替代方法 1 周          模型下载慢、进度显示不准
P3    体验问题             2 周          术语不清晰、布局不美观
P4    已知限制            不修           扫描件不支持、表格不结构化
```

### Failure 的分类箱

```
安装失败 (P0)
├─ install.sh exit code != 0
├─ NVIDIA 驱动加载失败
├─ Docker 安装失败
├─ GPU 容器验证失败
└─ 中国网络镜像不可用

检索失败 (P1)
├─ 用户说"没找到"
├─ 用户说"不对"
├─ 用户说"引用错了"
├─ 同一 query 两次结果不同
└─ benchmark 测试失败

文档失败 (P1)
├─ PDF 解析为空
├─ PDF 报错"扫描件"
├─ chunk 数量为 0
├─ chunk 内容乱码
└─ 上传后 status 一直 "processing"

体验失败 (P2)
├─ Dashboard 显示异常
├─ 模型安装进度不更新
├─ Chat 无响应
└─ 页面白屏
```

## 3. Feature Request Filtering

### 需求审查流程

```
用户说"我想要 X"
    │
    ▼
问题 1: 这个需求背后的问题是什么？
   ├─ 用户描述的是方案还是问题？
   ├─ 如果是方案，真正的问题是什么？
   └─ 例子: "我要 OCR" → 真实问题是 "我有扫描 PDF 打不开"
    │
    ▼
问题 2: 这个问题多久出现一次？
   ├─ 一次？→ 记下来，继续观察
   ├─ 三次？→ 可能是模式
   └─ 十次？→ 这是真需求
    │
    ▼
问题 3: 解决这个问题需要什么？
   ├─ 新的基础设施？→ 拒绝
   ├─ 新的 Python 依赖？→ 谨慎
   ├─ 不到 100 行代码？→ 考虑
   └─ 超过 500 行代码？→ 拒绝
    │
    ▼
问题 4: 添加这个功能后，系统会更复杂吗？
   ├─ 用户可见复杂度增加？→ 拒绝
   ├─ 维护负担增加？→ 拒绝
   ├─ benchmark 需要新增 10+ 测试？→ 可以接受
   └─ 用户核心路径不变？→ 可以接受
```

### 默认拒绝清单

```
以下需求，默认拒绝，除非有 10+ 用户要求:
□  云端同步
□  多用户/权限
□  移动端 APP
□  企业 SSO/LDAP
□  Agent 自动执行
□  工作流/审批
□  PDF 标注
□  Chat 历史持久化
□  自定义 embedding 模型
□  第三方集成（Slack/飞书/钉钉）
```

### 可以做的"改进"

```
以下改进，不需要用户请求也可以做:
□  错误提示更清晰
□  文档更友好
□  Benchmark 覆盖更全
□  安装兼容性更广
```

## 4. Retrieval Trust Monitoring

### 信任的观察指标

```
不信任 → 探索 → 接受 → 依赖

行为                             信任等级
────────────────────────────────────────
上传测试文件 "test.pdf"            ❌ 不信任
反复问同一个问题对比结果           ⚠️ 探索
展开 source snippet 验证           ⚠️ 探索
第一次说"嗯，这个对了"             ✅ 接受
上传真实敏感文件                   ✅ 接受
回答同事问题时引用系统结果         ✅ 依赖
不关 Dashboard                     ✅ 依赖
说"没有这个功能我就用不了"         ❌（其实是依赖）
```

### 信任危机的信号

```
□  用户说 "我点开了 source，发现它引用的内容不对"
    → 立即查检索 + chunker

□  用户说 "我问了三次，三次答案不一样"
    → 立即查 deterministic retrieval

□  用户说 "它编了一个条款"
    → 立即查 prompt + LLM 参数

□  用户说 "我回去用 ChatGPT 了"
    → 产品方向可能有问题
```

### 信任一旦破裂

```
修复信任比修复代码难得多。
一次错误的 citation 可能让专业用户永远不再回来。

所以:
  宁可说"不知道"，也不要编造。
  宁可返回空，也不要返回错误引用。
```

## 5. Complexity Drift Prevention

### Feature Explosion 的早期信号

```
信号                                                            阶段
──────────────────────────────────────────────────────────────────
团队开始说"这个功能不难"                                         早期
Issue 中出现 "顺便把 X 也做了"                                    早期
PR 中包含非必要的重构                                              中期
一个 PR 改动了超过 5 个文件                                        中期
用户提了一个需求 → 团队立即开始做                                 晚期
Roadmap 上有超过 3 个未确认的大功能                               晚期
代码行数在一个月内增长超过 20%                                     晚期
```

### 预防措施

```
1. 每个 PR 必须关联一个用户反馈（GitHub Issue 链接）
2. 每个功能必须经过 3-question 过滤
3. 每周 review benchmark 结果，而不是 review 新功能
4. 代码行数增长超过 10%/月 → 自动暂停新功能
5. 新增依赖必须删除旧依赖（一对一）
```

### v0.1.x 的代码规模目标

```
v0.1.0:   ~4400 行
v0.1.x:   <5000 行  （只允许 ~15% 的增长，全部用于 bug 修复）
v0.2.0:   <6000 行  （如果启动的话）
```

## 6. v0.2.0 Signal Detection

### 必须等待的信号

```
□  安装量 > 20 次（unique clones）
□  活跃用户 > 5 人（使用超过 1 周）
□  超过 5 个用户上传了真实专业文档
□  至少 3 个用户独立提出了同一个方向的需求
□  有用户说 "我愿意为功能 X 付费"
```

### 候选方向的触发门槛

```
OCR:          5+ 用户要求扫描件支持，且能提供测试文件
Advanced Retrieval: Benchmark 精度 < 80% 或 5+ 用户反馈检索问题
Multi-Doc:    3+ 用户要求对比多个文档
PDF Preview:  3+ 用户要求在 Dashboard 里看原文
Team:         5+ 用户要求同一台机器多人使用
```

### 如果 3 个月内没有信号

```
如果 3 个月后:
□  安装量 < 10
□  活跃用户 < 3
□  几乎没有用户反馈

那就说明:
  当前产品的定位或交付方式有问题。
  不应该启动 v0.2.0，而是应该重新审视产品方向。
```

## 总结

```
v0.1.0 发布 → 观察 → 等待信号 → 决定下一步

当前:
  34 个文件
  ~4400 行代码
  0 个待开发功能
  等待第一批真实用户

纪律:
  用户不反馈 → 不修改
  一个用户的问题 → 记下来继续观察
  三个用户同一问题 → 确认模式
  十个用户同一问题 → 立即修

方向:
  由真实使用场景驱动，不是由功能冲动驱动。
```
