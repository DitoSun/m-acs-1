# M-ACS-1 Observation Signal Taxonomy & Decision Pipeline

v0.1.0 真实用户观察阶段的完整信号分类体系与产品决策流程。

---

## 1. Signal Taxonomy（信号分类体系）

### 五大类，二十子类

```
TRUST SIGNALS (T)
├─ T1: 用户上传真实文件          → 隐私信任
├─ T2: 用户展开 source snippet   → 验证行为
├─ T3: 用户基于回答做决策        → 深度信任
├─ T4: 用户推荐给同事            → 社会信任
└─ T5: 用户持续使用超过 1 周     → 习惯形成

FRICTION SIGNALS (F)
├─ F1: 用户说 "没找到"           → 检索 miss
├─ F2: 用户说 "不对"             → 检索/理解错误
├─ F3: 用户重复同一 query        → 不信任结果
├─ F4: 用户问 "怎么用"           → 文档/UX 不足
└─ F5: 用户停止使用              → 信任破裂

DEPLOYMENT SIGNALS (D)
├─ D1: 安装一次性通过            → 兼容性 OK
├─ D2: 安装失败后重试成功        → 错误提示不足
├─ D3: 安装失败后放弃            → 严重问题
├─ D4: 用户问 "支持 X 环境吗"    → 平台盲区
└─ D5: 用户问 "怎么升级"         → 升级路径缺失

RETRIEVAL FAILURE SIGNALS (R)
├─ R1: Wrong chunk retrieved      → chunker/embedding 问题
├─ R2: Source citation wrong      → page mapping 错误
├─ R3: Cross-document confusion   → 多文档检索污染
├─ R4: Hallucinated content       → LLM 生成非原文内容
└─ R5: Empty retrieval            → 无相关 chunk 命中

WORKFLOW SIGNALS (W)
├─ W1: 用户上传多份文档          → 知识库需求
├─ W2: 用户问跨文档问题          → 多文档推理需求
├─ W3: 用户回访使用              → 产品粘性形成
├─ W4: 用户分享结果给他人        → 传播价值
└─ W5: 用户要求特定功能          → 方向信号（需过滤）
```

### 信号严重度

```
CRITICAL:   用户无法继续使用 -> P0
MAJOR:      用户功能受阻      -> P1
MINOR:      体验问题          -> P2
INFO:       可忽略            -> P3
POSITIVE:   系统工作正常      -> 存档
```

## 2. Real-World Retrieval Edge Cases

### Benchmark 覆盖不到的真实问题

```
Synthetic benchmark     Real-world document
────────────────────    ────────────────────────
标准 UTF-8 文本         政府系统导出的 GB18030 编码
规范的新行分割          银行对账单的混合换行符
单语言文档              中英日韩四混文档
人工设计的 query        用户随口问的模糊问题
fpdf2 标准生成 PDF      Word 转 PDF 的异常标签
单文档检索              10 份合同中找一句话
无表格/仅有文字         财报表格占 60% 页面
规范编号 (Article 1)    非标准编号 (A./I./Section A)
```

### 每个真实问题的检测方式

```
真实问题                    检测方式                     响应
──────────────────────────────────────────────────────────
GB18030 编码 PDF            解析后 avg < threshold      提示不支持编码
混合换行符                  文本包含 \r\n\r 等           自动归一化
多语言文档                  包含 CJK + Latin 混合        正常处理即可
模糊 query                  用户重复 3 次不同表达        无需修改（用户习惯）
Word 转 PDF 异常标签         提取文本含多余空格/标签       _normalize_text 缓解
10 份文档中检索 1 份        chunk 来自错误文档            file_ids 参数解决
表格密集页面                提取文本行数少/无结构         接受现状
非标准编号                  chunk 不按预期分割            chunker 新增模式
```

### 不可自动检测的（需要人工判断）

```
□  用户说 "不对" 但 system 认为对了
    → 需要人工分析 query 和预期

□  用户说 "找到了" 但 system 知道只匹配了 50%
    → 用户期望值低，不是系统好

□  用户上传了 PDF 但从未提问
    → 用户可能不知道可以提问，或文档处理失败未提示
```

## 3. Product Identity Reinforcement

### M-ACS-1 Identity 陈述

```
一句话:

  「一台本地运行的专业文档智能 Appliance。」

四根支柱:

  1. 本地 —— 文件不出本机，零云依赖
  2. 可信 —— 每个回答附带可验证的来源引用
  3. 低复杂度 —— 单机运行，一条命令安装，零基础设施
  4. 专业文档 —— 基于用户的 PDF 做智能分析，不是通用聊天
```

### 会直接破坏 Identity 的 Feature Request

```
请求                          破坏的支柱                    拒绝理由
──────────────────────────────────────────────────────────────
云端同步                      支柱 1（本地）                 变 SaaS
多用户/RBAC                  支柱 3（低复杂度）             10x 复杂度
Agent 自动执行                支柱 2（可信）                不可自动决策
通用聊天机器人                支柱 4（专业文档）             不是这个产品的方向
PDF 标注/批注                支柱 3（低复杂度）             PDF 渲染引擎
企业 SSO/LDAP                支柱 3（低复杂度）             不需要
移动端 APP                   支柱 3（低复杂度）             移动开发资源
知识库分享                    支柱 1（本地）                 需要云后端
```

### Identity 韧性测试

```
问: 加了这个功能后，用户还会说 "这是一台本机文档分析工具" 吗？
    如果不会 → 拒绝这个功能。

问: 加了这个功能后，系统还能在单台 Ubuntu 上运行吗？
    如果不能 → 拒绝这个功能。

问: 加了这个功能后，新用户能在 30 分钟内上手吗？
    如果不能 → 拒绝这个功能。
```

## 4. Professional Trust Indicators

### 信任阶梯

```
Step 1: 试探
  上传测试文件 → "试试看"
  不展开 source → "AI 说的应该差不多"
  信任度: 30%

Step 2: 验证
  上传真实文件 → "看看能不能找到我要的东西"
  展开 source snippet → "我得确认一下"
  信任度: 50%

Step 3: 接受
  基于回答做判断 → "合同里确实写了 20%"
  重复使用 → "再帮我查一个"
  信任度: 70%

Step 4: 依赖
  长时间不关 Dashboard → "随时可能要用"
  推荐给同事 → "你也可以装一下"
  信任度: 90%

Step 5: 捍卫
  对比 ChatGPT → "这个更可信，因为能看到原文"
  愿意付费 → "能不能加个功能，我愿意付钱"
  信任度: 100%
```

### 信任的量化指标

```
指标                        测量方式                   当前基线
────────────────────────────────────────────────────────────
上传真实文件比例            真实文件 / (真实+测试)      未知
Source 展开率              展开次数 / 回答次数         未知
复访率                      7 天内回访用户 / 总用户    未知
推荐率                      用户推荐次数 / 总用户      未知
回答接受率                  "对" / ("对"+"不对")       未知
```

### 信任一旦丧失

```
信任丧失的速度是建立的 10 倍。

一次错误的 citation → 用户可能再也不回来。
一次 LLM 编造 → 前功尽弃。

所以:
  宁可空，不可错。
  宁可 "文档中未提及"，不可编造。
```

## 5. Observation-to-Decision Pipeline

### 反馈 → 决策 流程

```
用户反馈进入
    │
    ▼
┌─ Step 1: 信号分类 ──────────────────┐
│  识别属于 5 大域中的哪个              │
│  T / F / D / R / W                  │
│  标注严重度: CRITICAL / MAJOR / ... │
└────────────┬──────────────────────────┘
             │
             ▼
┌─ Step 2: 频率计数 ──────────────────┐
│  第一次出现 → 记入观察日志            │
│  第二次出现 → 标注 "recurring"       │
│  第三次出现 → 升级为 "confirmed"     │
│                                     │
│  规则:                              │
│  1 次 = 记录, 不是行动信号          │
│  3 次 = 确认模式, 可以行动           │
│  10 次 = 普遍问题, 必须修            │
└────────────┬──────────────────────────┘
             │
             ▼
┌─ Step 3: Identity 过滤 ─────────────┐
│  检查是否破坏 4 根支柱               │
│  检查是否通过韧性测试                 │
│  如果会破坏 identity → 拒绝 + 说明   │
└────────────┬──────────────────────────┘
             │
             ▼
┌─ Step 4: 复杂度预算 ────────────────┐
│  估算代码行数                        │
│  < 100 行 → 可立即做                 │
│  100-500 行 → 需要计划               │
│  > 500 行 → v0.2.0 阶段考虑          │
│  检查是否超出 v0.1.x 上限 (5000 行)  │
└────────────┬──────────────────────────┘
             │
             ▼
┌─ Step 5: 决策 ──────────────────────┐
│  Accept: 立即修复                    │
│  Defer: 记录到 v0.2.0 候选清单       │
│  Reject: 解释原因并关闭              │
└──────────────────────────────────────┘
```

## 6. v0.1.x Stability Doctrine

### 什么叫 "正确演化"

```
v0.1.x 阶段的正确演化:
  └─ 更少的 Bug
  └─ 更清晰的文档
  └─ 更广的环境兼容
  └─ 更多的测试覆盖
  └─ 更低的用户困惑

v0.1.x 阶段的错误演化:
  └─ 更多的功能
  └─ 更多的配置选项
  └─ 更多的 UI 元素
  └─ 更多的依赖
  └─ 更多的基础设施
```

### 演化原则

```
原则 1: 修复优于增强
  修 Bug 永远是第一优先级。
  加功能是最后的选择。

原则 2: 删除优于添加
  如果有两种修复方式，选代码更少的那一种。
  如果新增了代码，寻找可以删除的旧代码。

原则 3: 文档优于代码
  用户困惑 → 先改进文档/错误提示。
  不要为了减少困惑而重新设计 UI。

原则 4: 测试优于实现
  发现新的边界情况 → 先加 benchmark 测试。
  然后修代码让测试通过。

原则 5: 等待优于猜测
  不确定用户是否需要某个改进 → 等更多信号。
  不提前做 "用户将来可能会需要" 的功能。
```

### v0.1.x 发布节奏

```
Patch 版本 (v0.1.1, v0.1.2, ...):
  触发条件: 发现 P0/P1 Bug
  内容:     仅 Bug 修复
  频率:     不固定，有 Bug 就发

维护版本 (v0.1.x):
  触发条件: 累计 3+ P2 改进
  内容:     Bug 修复 + 文档改进 + 测试扩展
  频率:     约每月 1 次

不做:
  v0.2.0 之前的任何新功能发布
```

### 最终测试

```
v0.1.x 的每个 patch 必须通过:
  □  benchmark.py → 全部 passed
  □  corpus.py --validate → 全部 passed
  □  至少一台真实 Ubuntu 机器安装验证
```
