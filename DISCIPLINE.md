# M-ACS-1 Long-Term Observation Discipline

v0.1.0 观察阶段的最终纪律框架。保护产品 identity，防止 feature creep。

---

## 1. Why "Third Occurrence" Is the Threshold

### 单个用户的问题，不一定是问题

```
第一次出现时:
  └─ 可能是那个用户独有的环境
  └─ 可能是用户操作错误
  └─ 可能是用户偏好，不是普遍需求
  └─ 可能是用户的网络/硬件问题

第二次出现时:
  └─ 可能只是巧合
  └─ 可能是同一类用户群体

第三次出现时:
  └─ 3 个独立用户，相同或类似反馈
  └─ 说明这不是巧合，是真实模式
  └─ 说明值得投入时间
```

### 为什么不是 1 次，也不是 10 次

```
1 次就行动 → feature panic
  今天用户 A 要 OCR → 加 OCR
  明天用户 B 要多用户 → 加多用户
  后天用户 C 要 Agent → 加 Agent
  → 3 个月后产品变成四不像
  而且每个功能只服务于一个人

3 次才行动 → disciplined evolution
  3 个独立用户要求 OCR → OCR 可能是真需求
  但不是立刻做，而是确认后放进候选清单
  然后等更多信号

10 次才行动 → 可能太晚了
  10 个用户都遇到了同一问题
  但可能已经流失了 5 个
  对于 P0 级别的问题，10 次是不可接受的

所以:
  P0/P1 (系统不可用/关键功能挂):  1 次就修
  P2/P3 (体验问题/优化):          3 次才确认
  Feature Request:                3 次才记录到候选清单
                                   10 次才考虑实现
```

### 例外情况

```
以下情况，1 次就行动:
  □  install.sh 在某个环境上崩溃
  □  LLM 编造不存在的内容
  □  用户的个人数据泄露（当前不可能，如果发生立即修）
  □  benchmark.py 完全不可通过

以下情况，永远不行动:
  □  用户说 "你能不能加一个 XX 功能"（单个用户）
  □  用户说 "我觉得应该这样做"（单个用户）
  □  用户说 "竞品有 XX"（不管多少个用户）
```

## 2. Signal Weighting Strategy

### 信号权重矩阵

```
信号类型                        权重  理由
────────────────────────────────────────────────────
Retrieval Wrong Citation       100  用户信任的根本
Retrieval Hallucination        100  绝对不可接受
Install Failure (P0)           100  无法开始使用
Retrieval Empty                80   功能受阻
PDF Parse Failure              80   无法处理文档
Source Page Wrong              70   引用不可信
Install Failure (network)      60   外部原因
Document Type Rejected         50   已知限制
Model Pull Timeout             40   网络问题
Dashboard Display Issue        30   体验问题
UI Terminology Confusion       20   可优化
Feature Request (single)       10   噪音
Feature Request (3+)           60   需确认
UI Preference                  5    个人偏好
```

### 加权后的决策阈值

```
累计权重 ≥ 100 → P0，立即修
  1 个 Retrieval Hallucination (100) → 立即修
  1 个 Install Failure (100) → 立即修

累计权重 ≥ 200 → P1，72h 内修
  3 个 Retrieval Wrong Citation (3×100=300) → 修
  2 个 Retrieval Empty + 1 个 PDF Parse Failure (2×80+80=240) → 修

累计权重 ≥ 300 → P2，下个 patch
  5 个 UI Terminology Confusion (5×20=100) → 记录
  3 个 Feature Request (3×60=180) → 记录到候选

累计权重 ≥ 500 → P3，观察
  10 个 Feature Request (10×60=600) → 考虑 v0.2.0
```

### 为什么权重比计数更准确

```
计数: 3 次 PDF 解析失败 = 3 次
  不能区分是同一个用户重复提交还是 3 个不同用户

加权: 3 个独立用户 × PDF 解析失败 (80×3) = 240
  更真实的反映了问题的影响范围
```

## 3. Identity Drift Detection

### 早期漂移信号

```
代码层面的漂移:
  □  新增了第一个外部 API 调用 → 开始依赖云
  □  新增了第一个用户表/登录 → 开始多用户
  □  新增了第一个后台任务/定时器 → 开始复杂化
  □  新增了第一个 WebSocket → 开始实时化
  □  Docker Compose 超过 3 个服务 → 开始分布式

功能层面的漂移:
  □  "能不能加个聊天记录" → 变成聊天产品
  □  "能不能分享给同事" → 变成协作平台
  □  "能不能自动审合同" → 变成 Agent 平台
  □  "能不能云备份" → 变成 SaaS

团队行为层面的漂移:
  □  开始说 "我们先搭个架构"
  □  开始说 "以后扩展方便"
  □  开始说 "这个重构一下以后好加功能"
  □  PR 描述中出现 "顺便把 X 也做了"
```

### Identity Drift 检查

```
每月一次 identity check:

□  系统还在单机运行吗？              Yes / No
□  文件处理还在本地吗？              Yes / No
□  每个回答还能追溯到原文吗？         Yes / No
□  一条命令还能安装吗？              Yes / No
□  没有引入新 Docker 服务？          Yes / No
□  没有引入新外部依赖？              Yes / No
□  没有用户系统？                     Yes / No

如果任何一个 No → 立即停止，重新评估方向
```

### 漂移恢复

```
如果发现 identity drift:

1. 立即停止导致 drift 的功能开发
2. 评估是否可以回滚
3. 如果可以回滚 → 回滚
4. 如果不能回滚 → 文档化 drift 原因
5. 在新版本中纠正方向
```

## 4. Complexity Budget Framework

### 量化指标

```
指标                    当前值        上限          超过上限行动
──────────────────────────────────────────────────────────────
代码行数 (LOC)          7675         9000          冻结新功能
Python 依赖             8            12            必须删除等价依赖
Docker 服务             3            3             不允许新增
运行时组件 (process)    3 (ollama,  4             必须合并或优化
                        control-plane,
                        host-agent)
配置文件                4            6             超过后合并
环境变量                8            12            超过后删除旧的
API 端点数              16           25            超过后合并或删除
数据库表                3            6             超过后必须优化 schema
```

### LOC 预算详细

```
v0.1.0:   7675 行 (基线)
v0.1.x:   <9000 行 (+17% 上限，全部用于 bug 修复和测试)
v0.2.0:   <12000 行 (如果启动)

每月检查:
  如果当月增长 > 10% → 冻结所有更改，只修 benchmark
  如果连续 3 个月增长 > 5% → 主动删除死代码
```

### 复杂度预算耗尽时

```
当预算耗尽时:
  1. 不提交新代码，直到有等价代码被删除
  2. 寻找可删除的死代码/重复代码/过度设计的代码
  3. 重构以减少行数
  4. 只有当 LOC 回到上限以下时才提交新代码
```

## 5. Professional Workflow Observation

### 最值得观察的行为模式

```
行为模式                       含义                        优先级
──────────────────────────────────────────────────────────────
用户上传文档后立即提问         用户知道怎么用               P0 观察
用户上传文档后不提问          用户可能不知道可以提问        P0 观察
用户上传多个同类文档           用户在做文档归档             P1 观察
用户上传不同类文档             用户在探索产品边界           P1 观察
用户反复问同一份文档           用户在深度分析               P1 观察
用户问跨文档问题               用户需求超出单文档           P2 观察
用户把结果分享给同事           用户认可回答质量             P2 观察
用户每天定时使用               用户形成了工作习惯           P2 记录
用户使用超过 30 分钟           用户深度使用                 P2 记录
用户同时使用 Chat + Docs      用户整合使用                 P3 记录
```

### Workflow Emergence 的判断

```
证明 workflow 形成的标准:
  □  同一用户 7 天内回访 ≥ 3 次
  □  每次回访上传新文档
  □  每次回访提出新问题
  □  用户行为模式可复现

如果满足以上 4 条 → 该用户的 workflow 已经固化
```

### 不应被视为 workflow 的行为

```
□  用户只安装了一次，再也没回来
□  用户只上传了测试文件
□  用户只问了一个问题
□  用户只在遇到问题时才打开
```

## 6. Long-Term Product Evolution

### v0.2.0 的信号门槛

```
硬性门槛（必须全部满足）:
  □  安装量 > 20
  □  活跃用户（7 天回访）> 5
  □  累计用户反馈 > 10 条

加权门槛（累计权重 ≥ 500）:
  □  同一个方向的 feature request 累计权重 ≥ 500
  □  或同一类 retrieval failure 累计权重 ≥ 300
```

### 候选方向的启动条件

```
方向                    启动条件                                      最低用户数
────────────────────────────────────────────────────────────────────────────
OCR/扫描支持             5 个独立用户要求 + 累计权重 ≥ 300              5
高级检索 (hybrid)        Benchmark 精度 < 80% + 累计权重 ≥ 300          N/A
多文档推理               3 个独立用户跨文档 query + 累计权重 ≥ 200       3
PDF 预览                 3 个独立用户要求 + 累计权重 ≥ 200              3
知识库管理               5 个用户上传 > 20 份文档                       5
API 扩展                 3 个独立用户要求集成                           3
团队共享                 5 个独立用户要求多用户                          5

注意: 条件满足 ≠ 立即启动。需要团队确认 identity 不受影响。
```

### 如果 6 个月后没有信号

```
情况 1: 安装量 < 10
  产品定位或交付方式有问题。
  不是加功能能解决的。需要重新审视。

情况 2: 安装量 > 10 但活跃用户 < 3
  首次体验有问题。
  用户安装了但没有持续使用的动力。

情况 3: 活跃用户 > 3 但信号权重 < 200
  用户可能只是浅尝辄止。
  也可能是用户对当前体验满意，没有更多需求。
  这两种情况的前进方向完全不同。

情况 4: 信号权重 > 500
  启动 v0.2.0 规划。
```

### 产品演化的底层原则

```
1. 不是每个问题都需要用代码解决
  用户困惑 → 先改文档/FAQ/错误提示
  检索不到 → 先确认文档中是否有该信息
  安装失败 → 先确认环境兼容性

2. 不是每个用户需求都应该被满足
  需求可能来自用户对产品的误解
  需求可能来自用户的独特环境
  需求可能导致 identity drift

3. 不是每个改进都应该立即做
  等第三次出现 → 确认这是个真问题
  评估复杂度 → 确认值得投入
  检查 identity → 确认不会漂移

4. 产品的核心资产不是功能多
  而是:
    本地部署的可信度
    单机运行的简单性
    来源引用的确定性
    专业用户的信任
```
