# First Professional User Observation — v0.1.0

观察第一批专业用户的真实使用行为，不做功能冲动。

---

## 1. First User Observation Priorities

### 优先级排序

```
P0（必须观察）         P1（值得观察）         P2（有了更好）
────────────────────   ────────────────       ────────────────
安装是否一次性通过      用户上传了什么文档     用户是否分享给同事
首次 model pull 结果   用户问了什么问题      用户是否回来使用
首次 document ask 结果 用户是否展开 source   用户是否改配置
用户是否理解 Dashboard 用户是否使用 Chat     用户是否看 FAQ
用户是否遇到错误       用户是否看 source     用户是否提交 Issue
```

### 每个用户的观察表

```
用户观察记录:
──────────────────────────────────────────
环境:           Ubuntu 24.04 / WSL2 / VM
GPU:            RTX 4090 / 4070 / 3090 / 其他
网络:           中国大陆 / 海外 / 内网
安装:           一次性通过 / 需要重启 / 失败
安装耗时:       多少分钟
首次模型:       Quick Start / 其他 / 未安装
上传文档类型:   合同 / 财报 / 法规 / 其他
首次 query:     什么类型的问题
检索结果:       找到 / 未找到 / 不准确
Source 使用:    展开 / 未展开
Chat 使用:      使用 / 未使用
整体体验:       满意 / 一般 / 不满意
具体问题:       [用户反馈]
```

### 不强求用户做的事

```
□  不强制用户注册/登录
□  不要求用户填写调查问卷
□  不追踪用户行为
□  不收集使用数据
□  用户可以选择完全不反馈
```

## 2. Failure Signal Ranking

### 优先级矩阵

```
严重性    信号                           影响                 处理优先级
──────────────────────────────────────────────────────────────────
Critical  LLM 编造不存在的条款            用户信任崩塌          P0
Critical  安装失败（非网络原因）          无法开始使用          P0
High      检索返回了错误的条款            用户找到错误信息      P1
High      source citation 页码不对       用户无法验证          P1
High      扫描 PDF 被拒但用户不理解为     用户体验差            P1
Medium   模型下载超时/失败                功能受阻              P2
Medium   Dashboard 显示异常              体验问题              P2
Low      界面术语不清晰                   可接受的困惑          P3
Low      PDF 表格文本混乱                 已知限制              P3
```

### P0 处理流程

```
Critical failure detected (LLM 编造 / 安装崩溃)
    │
    ▼
24h 内确认
    │
    ├─ 如果是 LLM 编造:
    │   ├─ 检查 prompt 约束强度
    │   ├─ 添加 explicit "禁止编造" 指令
    │   └─ 发布 patch 版本
    │
    └─ 如果是安装失败:
        ├─ 运行 collect-debug.sh
        ├─ 确定失败步骤
        └─ 发布 patch 版本
```

### Failure Signal 分类

```
fatal:     用户无法继续使用 → P0
blocking:  功能不可用 → P1
annoying:  体验不好 → P2
cosmetic:  不影响使用 → P3
```

## 3. Professional Trust Signals

### 用户开始信任的标志

```
信任等级    信号                         示例
──────────────────────────────────────────────────
L0: 怀疑    用户只上传测试文件             "先试试"
            用户反复验证引用              "让我点开看看"
            用户不展开 source             "AI 说的应该对吧"

L1: 接受    用户上传自己的真实文件         "这是我的审计报告"
            用户开始问真实问题             "今年净利润是多少"
            用户引用回答做决策             "合同显示违约金20%"

L2: 依赖    用户长时间不关 Dashboard      "一直开着"
            用户把文档分析结果发给同事     "你看这个条款"
            用户对比多个文档               "和上一版合同有什么不同"

L3: 信任    用户用系统替代 ChatGPT        "不用传文件给 ChatGPT 了"
            用户推荐给同事                 "你也装一个"
            用户愿意为功能付费             "能不能加 OCR"
```

### 信任的关键里程碑

```
P0 milestone:  用户上传了一份真实工作文档
P1 milestone:  用户基于检索结果做了一个真实决策
P2 milestone:  用户一周内回来了至少 3 次
P3 milestone:  用户说 "可以了，不需要加功能"
```

### 信任破裂的信号

```
□  用户说 "这个回答不对"
□  用户说 "我找不到我要的东西"
□  用户停止使用
□  用户回到 ChatGPT
```

## 4. Product Identity Protection

### 即使用户强烈要求，当前也不应该做

```
功能                       常见用户理由                   保护身份的理由
────────────────────────── ──────────────────────────── ─────────────────────────
云端同步                   "换电脑后数据怎么办"           需要云后端，变成 SaaS
多用户/RBAC               "我们团队都要用"               需要用户系统，复杂度 10x
Agent 自动审合同           "帮我自动审查风险"             法律场景不可自动决策
PDF 标注/批注             "我想在 PDF 上写意见"          需要 PDF 渲染引擎
移动端 APP                "我想在手机上查"               需要移动开发资源
自定义 embedding 训练      "行业术语需要定制"             需要训练数据，维护复杂度
企业 SSO/LDAP            "我们公司用企业账号"            单机场景不需要
工作流引擎                "合同审核需要两级审批"          偏离文档分析核心
```

### 当前可以说的话

> "M-ACS-1 的核心价值是 **本地、可信、低复杂度** 的专业文档理解。
> 我们不做大平台，也不做云服务。
> 
> 你提到的功能很有价值，但我们选择专注做好单机 Appliance 这一件事。
> 如果这些功能对你的团队是刚需，建议看看企业级的文档管理平台。
> 
> M-ACS-1 会持续改进检索精度和文档支持广度，
> 但我们不会变成一个全功能企业平台。"

### 保持 identity 的三条线

```
第一条线: 不引入新基础设施
  任何需要额外 Docker 容器、数据库、云服务的功能 → 拒绝

第二条线: 不引入用户系统
  任何需要注册、登录、权限、session 的功能 → 拒绝

第三条线: 不引入 AI 自动决策
  任何让 AI 代替人做决定的场景 → 拒绝（只做辅助分析）
```

## 5. v0.1.x Stability Plan

### 发布后唯一应该做的事

```
v0.1.x 阶段只修三类东西:
────────────────────────────────────
1. Bug 修复
   - Install.sh 失败
   - Retrieval 错误
   - Source 引用不对
   - Dashboard 显示异常

2. 文档改进
   - FAQ 补充
   - README 更新
   - 错误提示优化

3. 兼容性
   - 更多 Ubuntu 版本测试
   - 更多 NVIDIA 驱动版本测试
   - 更多 Docker 版本测试
```

### 明确不做

```
v0.1.x 不做:
□  OCR
□  多文档推理
□  高级检索
□  协作功能
□  UI 重构
□  性能优化（除非有明确瓶颈）
□  新 embedding 模型
□  新向量数据库
□  新文档格式支持
□  新 Dashboard 页面
```

### Patch 版本策略

```
v0.1.0 → v0.1.1
  如果出现: install.sh 在某个 Ubuntu 版本上失败
  只修: install.sh
  不打包: 任何新功能

v0.1.0 → v0.1.2
  如果出现: 检索精度明显问题
  只修: chunker / retriever / parser
  不打包: 任何新功能
```

## 6. v0.2.0 Trigger Conditions

### 进入 v0.2.0 的前提条件

**需要满足 3 个以上：**

```
□  超过 10 个用户完成安装并使用 Dashboard
□  超过 5 个用户上传了真实工作文档
□  超过 3 个用户反馈了同一类检索问题
□  首次 user 到 10 个 user 的时间 < 4 周
□  至少 1 个用户说 "我愿意付费"
```

### 候选方向的触发信号

| 方向 | 触发信号 | 最低用户数 |
|---|---|---|
| **OCR（扫描 PDF 支持）** | 5+ 用户要求扫描件支持，且都能提供测试文件 | 5 |
| **高级检索（hybrid / rerank）** | Benchmark 显示检索精度 < 80% | 10 |
| **多文档推理** | 3+ 用户要求 "对比两份合同" | 3 |
| **文档预览（PDF viewer）** | 3+ 用户要求 "在 Dashboard 里看原文" | 3 |
| **知识库管理** | 3+ 用户想分类/标签/整理文档 | 3 |
| **团队共享** | 5+ 用户要求同一台机器多账号 | 5 |

### 不应该启动 v0.2.0 的信号

```
□  "这个技术很新，应该加"
□  "竞品有这个功能"
□  "投资人说应该做"
□  "KOL 建议加"
□  "一个人重复提了 10 遍"
□  "我觉得用户会喜欢"
```

### 决策原则

```
v0.1.x: 只修 bug + 改文档 + 加测试
v0.2.0: 只有当真实用户数据明确指向一个方向
         且至少 3 个独立用户独立提出同一需求
         才启动

在 v0.2.0 之前:
  不写一行新功能代码。
```
