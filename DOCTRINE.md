# M-ACS-1 Stable Observation Doctrine

v0.1.0 已发布。项目进入产品存在状态。
当前最重要的能力不是 coding，而是 observation discipline。

---

## 1. Why Observation Discipline Is Now the Most Important Capability

### 代码已经完成了它的第一阶段使命

```
v0.1.0 证明了:
  □  可以在一条命令内完成 AI Server 部署
  □  可以在本地完成专业文档的理解
  □  可以在 8000 行代码内实现可信的 retrieval
  □  可以在不依赖云服务的情况下做到 source attribution

代码已经完成了"验证可行性"的使命。
现在代码的工作是: 稳定、修复、等待。
```

### 当前真正的产品工作是理解用户

```
代码能解决的问题:      安装失败、检索错误、页面显示异常
代码不能解决的问题:    用户是否信任检索结果
                      用户是否形成了使用习惯
                      用户真正需要什么
                      用户为什么不再回来

代码能回答 "what"。
只有 observation 能回答 "why"。
```

### 为什么 coding 不再是第一优先级

```
v0.1.0 的功能集已经足够回答一个重要问题:

  「专业用户是否愿意使用一个本地部署的文档分析工具？」

要回答这个问题，不需要更多功能。
需要的是: 观察、等待、理解。

如果答案是否定的:
  不是因为我们少了一个 feature。
  而是因为产品定位或交付方式有问题。

如果答案是肯定的:
  用户会告诉我们下一步该做什么。
```

### Observation 是第一优先级

```
Observation 不是 "不做事的借口"。
Observation 是当前阶段最重要的工作。

好的 observation 发现模式。
好的 observation 阻止 feature panic。
好的 observation 保护产品 identity。
好的 observation 确保下一次迭代有真实依据。
```

## 2. Signal Confirmation Framework

### 为什么重复出现比 loud feature request 更重要

```
Loud feature request:
  └─ 一个人说了 10 遍 → 仍然是一个人的需求
  └─ 用户在社交媒体上 rant → 仍然是一个人的需求
  └─ 用户是 KOL/投资人 → 仍然可能是一个人的需求

重复出现的真实模式:
  └─ 3 个独立用户遇到同一个问题 → 这是真问题
  └─ 3 个独立用户要求同一个功能 → 这是真需求
  └─ 3 个独立用户形成同一个 workflow → 这是真场景
```

### 信号的三个层次

```
第一层: 用户说了什么 (explicit)
  └─ 用户说 "我想要 OCR"
  └─ 权重: 10 (噪音)
  └─ 注意: 用户说的是方案，不是问题

第二层: 用户做了什么 (behavioral)
  └─ 用户上传了 5 份扫描 PDF
  └─ 权重: 50 (行为证据)
  └─ 注意: 行为比语言更真实

第三层: 用户反复做了什么 (pattern)
  └─ 用户连续 3 周每周上传扫描 PDF
  └─ 权重: 100 (确认模式)
  └─ 注意: 重复行为 = 真实工作流
```

### 1/3/10 规则的完整定义

```
1 次出现:
  └─ 记录到观察日志
  └─ 标注信号类型和权重
  └─ 不采取任何行动
  └─ 不讨论实现方案
  └─ 不估算工作量

3 次出现 (3 个独立用户):
  └─ 确认为真实模式
  └─ 记录到候选清单
  └─ 讨论实现方案（但不开始做）
  └─ 等待更多信号

10 次出现 (或累计权重 ≥ 500):
  └─ 确认为普遍需求
  └─ 进入产品决策流程
  └─ identity 检查 → 复杂度预算检查
  └─ 决定是否在 v0.2.0 实现

例外 (1 次就行动):
  └─ install.sh 崩溃 → P0
  └─ LLM 编造内容 → P0
  └─ benchmark 不可通过 → P0
  └─ 数据安全风险 → P0
```

### 权重计算示例

```
场景 A:
  3 个用户各提了一次 "想要 OCR"
  → 3 次 × Feature Request (10) = 30
  → ❌ 不足以行动

场景 B:
  3 个用户各上传了 5 份扫描 PDF
  → 3 次 × 行为证据 (50) = 150
  → ⚠️ 值得关注，记录到候选清单

场景 C:
  3 个用户每周上传扫描 PDF，持续 1 个月
  → 3 次 × 确认模式 (100) = 300
  → ✅ 这是真的 OCR 需求

场景 D:
  10 个用户上传了扫描 PDF
  → 10 次 × 行为证据 (50) = 500
  → ✅ 必须行动
```

## 3. Complexity Preservation Strategy

### 为什么这些预算重要

```
预算                   保护的资产                  突破后的后果
──────────────────────────────────────────────────────────────
LOC < 9000             可维护性                   代码失控，Bug 率上升
服务 ≤ 3               部署简单性                  运维成本 3x
依赖 ≤ 12              可重现构建                  依赖冲突，安全漏洞
运行时进程 ≤ 4          资源占用                    内存竞争，OOM
API 端点 ≤ 25          API 可理解性                API 文档复杂化
数据库表 ≤ 6            schema 可管理性             迁移困难
配置文件 ≤ 6           配置可理解性                 配置组合爆炸
环境变量 ≤ 12          部署确定性                   环境相关 Bug
```

### 哪些变化最容易突破预算

```
风险                       突破的预算            触发条件
──────────────────────────────────────────────────────────
引入 OCR                    LOC +1500            5 个用户要求
                                   依赖 +2 (PaddleOCR)
                                   服务 +0 (可内置)
引入多文档推理              LOC +500             3 个用户要求
                                   依赖 +0
                                   端点 +3
引入 PDF 预览               LOC +300             3 个用户要求
                                   依赖 +1 (PDF.js)
                                   服务 +0
引入知识库管理              LOC +400             5 个用户 20+ 文档
                                   依赖 +0
                                   端点 +5
引入多用户                  LOC +800             5 个用户要求
                                   依赖 +2 (session)
                                   服务 +1 (auth)
                                   端点 +10
                                   表 +3
```

### 预算保护机制

```
每月自动检查:
  □  LOC 是否超过 9000？
  □  服务是否超过 3 个？
  □  依赖是否超过 12 个？

如果超过:
  1. 冻结所有新功能开发
  2. 寻找可删除的代码/依赖
  3. 重构以降低复杂度
  4. 回到预算内后才解冻

例外:
  没有任何例外。预算就是预算。
```

## 4. Identity Protection Analysis

### M-ACS-1 的核心 identity

```
一句话:

  「一台本地运行的专业文档智能 Appliance。」

四根支柱:

  1. 本地优先 (Local-First)
     文件不出本机。不需要互联网。不需要云账号。

  2. 可信优先 (Trust-First)
     每个回答附带来源引用。用户可以验证原文。

  3. 低复杂度优先 (Simplicity-First)
     单机运行。一条命令安装。零基础设施。

  4. 文档优先 (Document-First)
     不是聊天机器人。不是通用知识库。
     是「基于你的文档」的智能分析。
```

### 最危险的 Feature Request

```
危险等级    方向                   破坏的支柱         伪装形式
──────────────────────────────────────────────────────────────
CRITICAL    云端同步               本地优先            "换电脑时数据怎么办"
CRITICAL    多用户/RBAC            低复杂度           "团队要用"
CRITICAL    Agent 自动执行         可信优先            "帮我自动审合同"
HIGH        PDF 标注/批注          低复杂度           "我想在 PDF 上写意见"
HIGH        移动端 APP             低复杂度           "我想在手机上查"
MEDIUM      OCR 识别               低复杂度           "我有很多扫描件"
MEDIUM      知识库分享             本地优先            "我想和同事共享"
LOW         更多 embedding 模型    低复杂度           "我想用更好的模型"
LOW         API 扩展               低复杂度           "我想集成到我的工具"
```

### Identity 保护的最终检查

```
当考虑任何一个新功能时:

  问 1: 这个功能会让用户不再说 "这是一台本机文档分析工具" 吗？
  如果是 → 拒绝

  问 2: 这个功能需要额外的服务/数据库/云依赖吗？
  如果是 → 拒绝

  问 3: 这个功能会让代码行数增加超过 20% 吗？
  如果是 → 拒绝

  问 4: 这个功能是 10 个独立用户要求的吗？
  如果不是 → 拒绝

  问 5: 这个功能不做的风险是什么？
  如果风险是 "用户可能用 ChatGPT 替代我们" →
    这不是产品问题，这是产品定位问题。
    不应该通过加功能来解决。
```

## 5. Professional Workflow Emergence

### 最值得观察的自然 workflow

```
Workflow 1: Document Archive (文档归档)
  用户上传文档 → 从未提问
  → 含义: 用户需要的是一个文档管理系统，不是分析工具
  → 信号: 不是产品方向的问题

Workflow 2: Question & Answer (问答)
  用户上传文档 → 提问 1-3 个问题 → 离开
  → 含义: 用户把系统当作"文档版搜索引擎"
  → 信号: 核心场景，需要提升检索精度

Workflow 3: Deep Analysis (深度分析)
  用户上传文档 → 提问 10+ 个问题 → 持续回访
  → 含义: 用户在做真正的专业知识工作
  → 信号: 验证产品定位正确

Workflow 4: Cross-Document Analysis (跨文档分析)
  用户上传多份文档 → 问跨文档问题
  → 含义: 用户需求超过单文档检索
  → 信号: 多文档推理需求的证据

Workflow 5: Collaborative Verification (协作验证)
  用户检索 → 展开 source → 截图发给同事
  → 含义: 用户信任检索结果，并用于实际工作
  → 信号: 用户真正依赖产品
```

### 不应被视为 workflow 的行为

```
□  只安装了一次，再也没回来 → 安装体验没问题但价值不够
□  只上传了测试文件 → 用户还没信任系统
□  只问了一个问题 → 用户只是好奇
□  只在出问题时才打开 → 用户没有形成使用习惯
□  问完问题不看 source → 用户不关心准确度
```

### Workflow 固化判定

```
一个 workflow 固化的判定标准:
  □  同一用户 7 天内回访 ≥ 3 次
  □  每次回访上传新文档或提出新问题
  □  持续超过 2 周
  □  每次使用时长相较前一次不减

如果满足以上 → 该用户的 workflow 已经固化。
产品定位被验证。
```

## 6. Long-Term Evolution Philosophy

### M-ACS-1 的演化原则

```
演化由真实信号驱动，不是由功能冲动驱动。
演化由用户行为驱动，不是由竞品动向驱动。
演化由 discipline 驱动，不是由 FOMO 驱动。

演化是:
  当足够多的独立用户重复同一个行为后，
  基于 identity 保护和复杂度预算的审慎回应。

演化不是:
  对 loud feature request 的即时响应。
  对 AI 技术趋势的追赶。
  对竞品功能的复制。
```

### 产品生命周期

```
Phase 0 (已完成): 验证可行性
  目标: AI Server 能否在一条命令内部署？
  产出: install.sh + Dashboard

Phase 1 (已完成): 验证可用性
  目标: 用户能否通过 UI 完成基本操作？
  产出: 模型管理 + Chat

Phase 2 (已完成): 验证价值
  目标: 专业文档是否能被本地 AI 理解？
  产出: Document Pipeline + Workspace

Phase 2.5 (当前): 观察
  目标: 真实专业用户是否愿意使用？
  活动: 观察、修复、等待

Phase 3 (未来): 演化
  触发条件: 真实用户信号累计权重 ≥ 500
  方向: 由信号决定，不是由规划决定
```

### v0.2.0 的启动条件

```
硬性门槛 (AND):
  □  安装量 > 20
  □  活跃用户（7 天回访）> 5
  □  累计反馈 > 10 条

加权门槛 (AND):
  □  同一方向累计权重 ≥ 500
  □  identity 检查通过
  □  复杂度预算有余量

如果条件未满足:
  继续观察。
  不启动 v0.2.0。
  不提前写代码。
```

### 最终信念

```
M-ACS-1 最宝贵的资产不是 8048 行代码。
而是:

  □  本地优先的信任
  □  可信 retrieval 的确定性
  □  低复杂度的可维护性
  □  专业文档的专注

这些东西不是加功能能获得的。
它们是保持 discipline 的结果。

所以:

  停止扩张。
  开始观察。

  真实专业用户会告诉我们:
  哪些东西值得变复杂。

  而不是相反。
```
