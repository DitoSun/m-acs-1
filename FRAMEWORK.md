# M-ACS-1 Observation Signal Framework

v0.1.0 真实用户观察阶段的信号分类与分析框架。

---

## 1. Observation Signal Framework

### 五大信号域

```
域 1: Retrieval Trust       ← 检索结果是否可信
域 2: Deployment Friction   ← 安装部署是否顺畅
域 3: Document Robustness   ← 文档处理是否稳定
域 4: Source Verification   ← 引用溯源是否可靠
域 5: Workflow Emergence    ← 用户是否形成使用习惯
```

### 信号分类卡

```
信号卡格式:
──────────────────────────────────────────
域:        [域名称]
信号:      [具体用户行为或反馈]
等级:      ✅ 正面 / ⚠️ 注意 / ❌ 负面
频率:      单次 / 多次 / 持续
影响:      这个信号意味着什么
响应:      应该如何回应
──────────────────────────────────────────
```

### 信号收集渠道

```
收集渠道                用途                      优先级
──────────────────────────────────────────────────────────
GitHub Issues           结构化 Bug 报告             P0
collect-debug.sh 输出    安装/环境问题              P0
README / FAQ 页面访问    用户自学行为               P1
用户直接反馈（邮件/IM）   非结构化反馈               P1
FEEDBACK.md 提交         深度使用体验               P2
GitHub Stars / Watch     社区兴趣                   P3
```

## 2. Signal Cards (Pre-defined)

### 域 1: Retrieval Trust

```
信号 1.1
  信号:      用户上传真实工作文档
  等级:      ✅ 正面
  含义:      用户信任系统不会泄漏文件
  下一步:    不用做什么

信号 1.2
  信号:      用户说 "没找到"
  等级:      ⚠️ 注意
  含义:      检索可能 miss，或者 query 表达不准
  下一步:    确认文档中是否确实有该信息

信号 1.3
  信号:      用户说 "回答不对"
  等级:      ❌ 负面
  含义:      检索到错误条款或 LLM 理解错误
  下一步:    复现 query，检查 chunk 命中

信号 1.4
  信号:      用户重复同一个 query 多次
  等级:      ⚠️ 注意
  含义:      用户不信任结果，在验证
  下一步:    确认 deterministic retrieval
```

### 域 2: Deployment Friction

```
信号 2.1
  信号:      install.sh 一次性通过
  等级:      ✅ 正面
  含义:      环境兼容性好

信号 2.2
  信号:      install.sh 失败后用户重试成功
  等级:      ⚠️ 注意
  含义:      错误提示可能不够清晰
  下一步:    检查 collect-debug.sh 输出

信号 2.3
  信号:      install.sh 失败后用户放弃
  等级:      ❌ 负面
  含义:      安装体验有严重问题
  下一步:    立即复现并修复

信号 2.4
  信号:      用户问 "支持 Windows 吗"
  等级:      ⚠️ 注意
  含义:      README 可能没写清楚
  下一步:    检查 README 平台说明
```

### 域 3: Document Robustness

```
信号 3.1
  信号:      用户上传 PDF 后 status 一直是 processing
  等级:      ❌ 负面
  含义:      PDF 解析失败或 embedding 超时
  下一步:    获取该 PDF 测试

信号 3.2
  信号:      用户上传的 PDF 被提示 "扫描件"
  等级:      ⚠️ 注意
  含义:      用户遇到了不支持的类型
  下一步:    确认是否真的是扫描件

信号 3.3
  信号:      用户上传了 >100 页的 PDF
  等级:      ⚠️ 注意
  含义:      长文档处理是否符合预期
  下一步:    检查处理时间和 chunk 数量

信号 3.4
  信号:      用户上传了同一份文件多次
  等级:      ✅ 正面（去重生效）
  含义:      SHA256 去重正常工作
```

### 域 4: Source Verification

```
信号 4.1
  信号:      用户展开 source snippet 查看
  等级:      ✅ 正面
  含义:      用户在使用 source verification

信号 4.2
  信号:      用户说 "你引用的页码不对"
  等级:      ❌ 负面
  含义:      Page mapping 有偏移
  下一步:    立即复现

信号 4.3
  信号:      用户说 "这个引用是错的"
  等级:      ❌ 负面
  含义:      chunk 内容与原文不匹配
  下一步:    获取 PDF + query 复现

信号 4.4
  信号:      用户基于回答做了决策
  等级:      ✅ 正面（高风险）
  含义:      用户信任 source → 但责任也在我们
```

### 域 5: Workflow Emergence

```
信号 5.1
  信号:      用户第二天再次打开 Dashboard
  等级:      ✅ 正面
  含义:      用户觉得有用，愿意回来

信号 5.2
  信号:      用户上传了多个文件（超过 3 个）
  等级:      ✅ 正面
  含义:      用户开始积累知识库

信号 5.3
  信号:      用户问跨文档的问题
  等级:      ✅ 正面
  含义:      用户的需求已经超过单文档检索

信号 5.4
  信号:      用户把回答截图发给同事
  等级:      ✅ 正面
  含义:      用户认可回答质量，愿意传播
```

## 3. Failure Analysis Cards

### 真实用户环境中才可能出现的 failure

```
Benchmark 覆盖不到的真实问题:
──────────────────────────────────────────
FA-1:   PDF 字体编码特殊（非标准 Unicode）
        → benchmark 用的标准 PDF
        → 真实用户可能上传政府/银行系统导出的畸形 PDF

FA-2:   文档中有大量 OCR 残留字符
        → benchmark 用的合成 PDF
        → 真实文档可能经过扫描+OCR，满篇错别字

FA-3:   用户 query 表达不精确
        → benchmark 用的是设计好的 query
        → 真实用户可能问 "这个东西多少钱" 而不是 "Purchase Price"

FA-4:   跨文档混淆
        → benchmark 单文档测试
        → 真实用户可能上传 10 份文档，问其中 A 的内容，B 的内容被误召回

FA-5:   多语言混合文档
        → benchmark 中英文分开
        → 真实文档可能一段中文一段英文

FA-6:   不同 PDF 生成工具的质量差异
        → benchmark 用 fpdf2 / PyMuPDF 生成
        → 真实 PDF 可能来自 Word / WPS / 扫描仪 / 银行系统
```

### Failure 响应模板

```markdown
## Retrieval Failure Report

用户 ID:        [匿名 ID]
PDF 类型:       合同 / 财报 / 法规 / 其他
PDF 来源:       Word / 扫描 / 系统导出
PDF 页数:       [N]
Query:          "[用户的问题]"
预期回答:       简短描述
实际结果:       简短描述

Benchmark 是否覆盖: Yes / No
是否可复现:       Yes / No
根因:            chunker / embedding / prompt / 其他
修复:            [修复内容]
```

## 4. Identity Preservation

### M-ACS-1 的 identity

```
M-ACS-1 是一台「本地专业文档智能 Appliance」。

它的核心 identity 由以下四根支柱支撑:
──────────────────────────────────────────

支柱 1: 本地
  文件不出本机。
  不需要互联网。
  不需要云账号。

支柱 2: 可信
  每个回答附带来源引用。
  用户可以验证原文。
  不编造（prompt 约束）。

支柱 3: 低复杂度
  单机运行。
  零额外基础设施。
  一条命令安装。

支柱 4: 专业文档
  不是聊天机器人。
  不是通用知识库。
  是「基于你的文档」的智能分析。
```

### 绝对不能失去的

```diff
+  绝对不能失去: 本地部署（支柱 1）
+  绝对不能失去: 来源引用（支柱 2）
+  绝对不能失去: 单机运行（支柱 3）
+  绝对不能失去: 文档为中心（支柱 4）

-  可以失去: 某个特定格式支持
-  可以失去: 某个 UI 布局
-  可以失去: 某个 embedding 模型
-  可以失去: 某个 chunking 策略（可以换）
```

### Identity 检查三问

```
当我们考虑是否做一个功能时:

1. 这个功能会破坏「本地」吗？
   如果是 → 不做

2. 这个功能会降低「可信」吗？
   如果是 → 不做（或者必须附带更强的验证机制）

3. 这个功能会增加多少复杂度？
   如果 > 20% 代码增长 → 不做
```

## 5. Complexity Pressure Monitoring

### 复杂度压力的早期信号

```
代码层面的信号:
  □  单次 PR 改动超过 5 个文件
  □  新增依赖（pip install）
  □  新增 Docker 容器
  □  新增配置文件
  □  新增环境变量

团队行为层面的信号:
  □  开始说 "这个重构一下"
  □  开始说 "先把架构搭好"
  □  开始说 "以后扩展需要"
  □  开始计划 "v2 架构"
  □  开始引入设计模式

用户反馈层面的信号:
  □  用户开始要求 "集成到我们的系统"
  □  用户开始要求 "更多的配置选项"
  □  用户开始要求 "支持更多文件格式"
  □  用户开始要求 "API 限流/鉴权"
```

### 复杂度预算

```
当前基线:  ~4500 行（v0.1.0）
v0.1.x 上限: <5000 行
超出上限: 必须删除等价代码

每新增 100 行:
  必须找到 100 行无用的/重复的/过时的代码删除
```

### Feature Request 的复杂度评估

```
需求             预估行数    新依赖?   新基础设施?   接受?
────────────────────────────────────────────────────────────
OCR              800+       ✅ PaddleOCR  ❌            ❌
多文档对比        200+       ❌          ❌            ⚠️ 待定
PDF 预览          300+       ✅ PDF.js    ❌            ⚠️ 待定
API 鉴权          100+       ❌          ❌            ❌（不需要）
多用户            500+       ❌          ❌            ❌（不需要）
表格结构化提取    400+       ✅ 新模型    ❌            ❌
更好的错误提示     20+        ❌          ❌            ✅ 随时可做
```

## 6. Long-Term Evolution Signals

### 进入下一阶段的信号条件

```
v0.1.0 → v0.2.0 的门槛:    安装量 > 20 + 活跃用户 > 5 + 3 个独立信号
v0.2.0 → v0.3.0 的门槛:    安装量 > 100 + 活跃用户 > 20 + 10 个独立信号
v0.3.0 → v1.0.0 的门槛:    安装量 > 1000 + 活跃用户 > 100 + 付费用户 > 10
```

### 能力扩展的触发信号

```
方向                  触发条件                            最低用户数
──────────────────────────────────────────────────────────────────────
OCR 支持              用户反复要求扫描件，且能提供测试样本    5
多文档推理            用户反复问 "对比这两份"                 3
高级检索              基准测试精度 < 80%                      10
PDF 预览              用户要求 "在 Dashboard 看原文"          3
知识库管理            用户上传文档超过 20 份                  5
API 扩展              用户要求 "集成到我的工具"              5
工作流                用户要求 "自动处理"                     3
团队版                用户要求 "给团队用"                     5
```

### 如果信号没有出现

```
3 个月后检查:

情况 A: 安装量 < 10
  产品定位或交付方式有问题。
  不是加功能能解决的。

情况 B: 安装量 > 10 但活跃用户 < 3
  首次体验有问题。
  用户安装了但没有持续使用的动力。

情况 C: 活跃用户 > 3 但没人提任何需求
  用户可能只是浅尝辄止，没有深度使用。
  也可能是用户对当前体验满意。

情况 D: 活跃用户 > 3 且有人提需求
  这是最好的情况。
  基于需求决定下一步。
```

## 总结

```
当前阶段:  观察
目标:      等待真实用户信号
不做:      任何新功能

信号框架:  5 大域 × 每个域 4-5 个信号
Failure 分析: 6 种 benchmark 覆盖不到的真实问题
Identity:     4 根支柱（本地/可信/低复杂度/专业文档）
复杂度预算:   基线 4500 行，上限 5000 行
演化条件:     基于信号和数据，不是基于冲动
```
